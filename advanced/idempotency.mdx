---
title: "Prevent double charges with idempotent API requests"
description: "Learn how Velobase prevents double charges when network failures force a retry, and which identifier field to use for each billing endpoint."
---

Network requests can fail after Velobase has already processed them — a dropped connection returns no response, but the balance has moved. Without idempotency, retrying that request would charge the customer twice. Velobase uses per-endpoint identifier fields to detect duplicate requests and return the original result instead of re-processing the operation.

## How idempotency works

When Velobase receives a billing request, it checks whether the provided identifier has already been processed for that customer.

<Steps>
  <Step title="First request">
    Velobase processes the operation, updates the customer's balance, and returns the result with `"is_idempotent_replay": false`.
  </Step>
  <Step title="Duplicate request">
    Velobase recognizes the identifier, skips the operation entirely, and returns the exact same response as the original — this time with `"is_idempotent_replay": true`.
  </Step>
</Steps>

Your code can safely retry any billing request on timeout or network error. If the first attempt succeeded, the duplicate is a no-op that returns the same data. If the first attempt never reached Velobase, the retry is processed normally.

## Identifier fields by endpoint

Velobase uses different identifier fields depending on the operation. Pass the identifier in the JSON request body.

| Endpoint | Identifier field | Description |
|---|---|---|
| `POST /v1/billing/deposit` | `idempotency_key` | A unique string generated by your system for this specific deposit attempt. |
| `POST /v1/billing/deduct` | `transaction_id` | The unique ID of the task or job in your system that triggered the billing event. |
| `POST /v1/billing/freeze` | `transaction_id` | The unique ID of the task or job being staged for deduction. |
| `POST /v1/billing/consume` | `transaction_id` | Must match the `transaction_id` used in the corresponding freeze. |
| `POST /v1/billing/unfreeze` | `transaction_id` | Must match the `transaction_id` used in the corresponding freeze. |

## The `transaction_conflict` error

Identifiers are scoped to the `customer_id` they were first used with. If you send the same `transaction_id` or `idempotency_key` for a **different** customer, Velobase rejects the request with a `409 Conflict`:

```json
{
  "error": {
    "message": "transactionId already belongs to another customer",
    "type": "conflict",
    "code": "transaction_conflict"
  }
}
```

This error means the identifier is already owned by another customer in your system. Generate a new, unique identifier and retry.

## Best practices for generating identifiers

<Tip>
Use UUIDv4 for all billing identifiers. Most languages have a built-in or widely-used library: `crypto.randomUUID()` in Node.js, `uuid.uuid4()` in Python, `Uuid::new_v4()` in Rust. If you prefer human-readable keys, composite strings like `user-123-task-456` work well — just ensure the combination is globally unique across all customers in your system.
</Tip>

Avoid reusing identifiers across different logical operations. A `transaction_id` should map one-to-one with a single billable event in your system — not to a session, a user, or a job type.
